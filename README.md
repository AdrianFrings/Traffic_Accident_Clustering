# Traffic Accident Clustering Project
Identifying High-Leverage Intervention Points for Traffic Safety using Machine Learning and Geospatial Analytics.**

## The Problem
Across Germany, around 250k traffic accidents with human injury occur every year. Accidents are among the leading causes of human fatalities, especially for younger people. Making our streets and traffic safer is therefore a meaningful endeavor.

For Germany, the so-called Unfallatlas (https://unfallatlas.statistikportal.de/), literally meaning accident atlas, is a meaningful data source, mapping out all accidents that occurred in a single year with a geolocation on a map (if they included personal injury). This is a great resource to see where accidents occur, but it is hard to make out where accidents happen systemically over years, as only one year can be seen at a time. It is also hard to make out which would be the highest impact spots to improve accident mitigation outcomes for city planners or municipal governments.

## The Solution: Targeted Geospatial Intelligence
To solve this, we built the Accident Risk Dashboard. It displays the 3-8 highest density accident clusters for a city of choice in 3 categories (car, bicycle, and pedestrian injury). This allows for a targeted view on intersections that have meaningful potential for harm reduction.

## Technical Architecture
The dashboard is a full-stack, Dockerized data pipeline and interactive geospatial web application that digests nearly a decade of German traffic accident data (2016–2024), spanning over 3.2 million distinct accidents.

It utilizes the DBSCAN clustering algorithm to identify and distill these millions of scattered data points into geographic hotspots, segregated by transport mode (Bicycle, Car, and Pedestrian). 

The DBSCAN algorithm stands for Density-Based Spatial Clustering of Applications with Noise. Unlike other clustering algorithms (like K-Means) that require you to pre-define the number of clusters you want to find, DBSCAN discovers them organically based on the actual geographic grouping of the data. It works by evaluating a single accident point and drawing a search radius around it. If it finds a minimum number of other accidents within that radius, it forms a "core" cluster. It then expands outward, checking the neighbors of those neighbors to grow the cluster boundary. Crucially, any isolated accidents that do not meet this density threshold are categorized as 'noise' and ignored. This makes it the ideal algorithm for our use case, as it naturally filters out random, one-off crashes and isolates only the true, systemic danger zones.

We set the \`eps\` (search radius) value to 50 meters, identifying two given points as a related section of a street. The logic behind 50 meters is that two accidents that happen within 50 meters of each other are likely to belong to the same street setting. The minimum amount of points for the clusters is variable. Applying the algorithm across the entire country with the same minimum would be difficult. An algorithm tuned to find clusters in rural towns would generate a single, massive cluster covering an entire large city, and vice versa. 

To overcome this, we implemented dynamic density scaling with a loop:
We set the system to iterate through all 2,000+ municipalities one by one. For each city, the algorithm runs a continuous loop, dynamically scaling up the minimum required accident density (\`min_samples\`) until it actively suppresses the background noise and isolates the top 3-to-8 most accident-dense clusters.

### Why DBSCAN instead of HDBSCAN?
A common modern alternative to DBSCAN is HDBSCAN, which natively solves the problem of varying density without needing a manual loop. However, we deliberately chose the classic DBSCAN because it allows us to strictly enforce the \`eps\` parameter. 

In HDBSCAN, the spatial radius is flexible, meaning a cluster could theoretically stretch for several hundred meters along a sparsely populated rural road. While mathematically valid, this is practically useless for a city planner looking for specific, high-leverage intervention points. By hardcoding DBSCAN’s \`eps\` to exactly 50 meters, we force the algorithm to obey physical reality: a cluster *must* fit within the typical footprint of a single intersection or a tight street segment. We then handle the varying regional densities not by relaxing the spatial radius, but by dynamically scaling the \`min_samples\` threshold per municipality. This ensures every resulting hotspot is tightly localized and highly actionable.

## Data Distillation
Using the PyArrow engine, exactly 3.2 million individual raw crash coordinates are preserved, reverse-geocoded to street names, enriched with spatial jitter metrics, and exported as Parquet files. For the frontend, we use a simple containerized web application built on Streamlit and with Folium maps.

## Technologies/Packages Used
* **Data Pipeline:** Pandas, GeoPandas, Scikit-Learn (DBSCAN), Pyproj.
* **Frontend UI:** Streamlit, Streamlit-Folium, Altair.
* **Geospatial Rendering:** Matplotlib, Contextily, Reverse Geocoder.
* **Deployment Infrastructure:** Docker container running on a VPS.
